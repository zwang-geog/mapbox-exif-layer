Two layer classes supplied by this package

Smooth raster layer

A.k.a. sample fill in windgl, colorize in wind-layer — a different way to render classic raster data in the browser. Each grid cell holds one or more bands (e.g. temperature) and has a fixed size (¼°, 5 km, 500 m, etc.), so it looks like a box. The usual web approach is to color each cell, serve tiles, and get a blocky layer like QGIS at native resolution. Weather fields are spatially smooth, so we want a softer display. With WebGL varyings and a fragment shader, colors interpolate across space on the client (WebGL fundamentals) without hand-built downsampling. (Note: to achieve a good smoothing effect, using JPEG/PNG image source instead of GeoTIFF is recommended.)

Particle motion layer

The companion approach for vector fields such as wind, where each cell stores u- and v-component velocity rather than a single scalar. A conventional map might use static arrows. A particle layer releases hundreds of thousands of short-lived points (with tails); each frame, every particle samples local u/v, moves a small step, and is recolored by speed. The result is a flowing animation that makes direction and relative speed easy to read. Updates run on the GPU (transform feedback and a dedicated update vertex shader), integrated as a Mapbox/MapLibre custom layer so pan and zoom stay in sync with the basemap and other layers.

Frequently asked questions

When should I use mapbox-exif-layer?

  • Map wind as a dynamic particle motion layer.
  • Map weather (temperature, humidity, precipitation) — or any scalar field — where you want a smooth, non-blocky look via client-side linear interpolation (best with JPEG/PNG image sources).
  • Visualize a small GeoTIFF in the browser without setting up a raster tile server.

How does it differ from other wind-mapping solutions?

  • Low-cost processing and hosting: upload a properly encoded JPEG/PNG image to a public S3 bucket, your site's static directory, or similar — little infrastructure and ongoing cost.
  • Mobile browser compatible: built on the Mapbox/MapLibre custom-layer interface; wherever those SDKs run on a phone or tablet browser, these layers can too.
  • Respects actual u/v velocity: particles move according to local u- and v-component values, not at one uniform speed across the map.
  • Speed-colored particles: particle color reflects wind speed in the same layer — no separate speed raster to stack on top — so wind overlays cleanly with other weather layers such as temperature.
  • Avoids degenerate particle patterns: particle age is tracked on the GPU and stale particles are reset across the extent — reducing the “locked-in” cyclonic or atmospheric-river loops that some wind visualizations show when particles never leave large-scale flow features.

How are JPEG/PNG image sources encoded?

A JPEG or PNG used as a source is not a normal photo — it stores a normalized weather grid in three 8-bit RGB bands (0–255 per channel). For wind (ParticleMotion), u-velocity is min–max normalized to 0–255 in the R band and v-velocity in the G band; no-data cells use B = 0 (valid cells use B = 255). For a smooth raster (SmoothRaster), the scalar attribute is normalized to 0–255 in the R band; no-data cells use B = 255 (valid cells use B = 0).

The layer also needs the physical min and max values used when encoding (before normalization). You can supply them in either of two ways:

  • EXIF — write min/max to JPEG ImageDescription (the classic “EXIF-enabled JPEG” path; see Method 2 in the docs).
  • Constructor (v1.3.2+) — pass scalarValueRange on SmoothRaster or velocityRange on ParticleMotion when the image has no EXIF metadata (Method 1). If EXIF is present, it takes precedence.

See docs/jpeg-source.md for encoding rules, both min/max workflows, and ready-to-use pipeline scripts.

When should I use a JPEG/PNG image source vs GeoTIFF?

Properly encoded JPEG/PNG rasters are much smaller than GeoTIFFs. On this site’s NOAA NAM NEST CONUS demo grid (2269×976), wind is ~332 KB as JPEG vs ~8.9 MB as ZSTD-compressed float32 GeoTIFF — a big difference for storage and download when you serve many forecast timesteps.

  • Prefer JPEG/PNG when you need many timesteps (multi-hour forecasts), want minimal hosting cost, or want the smoothest SmoothRaster look (linear texture filtering on 8-bit RGB). Min/max metadata can come from EXIF or constructor scalarValueRange / velocityRange.
  • Prefer GeoTIFF (EPSG:4326, Float32, compressed) when you want a GIS-friendly pipeline without custom image encoding, have a smaller study area, or only a few timesteps. Extra float precision is often not visible in everyday visualization.

Can this package be used with Leaflet or OpenLayers?

No. It targets Mapbox GL JS and MapLibre GL JS only — both WebGL map SDKs with a similar custom-layer API that this package implements. Leaflet and OpenLayers use a different rendering model and are not supported.

Getting started

This package does not include a map SDK. Install one map runtime, then add mapbox-exif-layer (and optionally geotiff for .tif sources).

Mapbox GL JS

Default mapRuntime: 'mapbox'

npm install mapbox-gl

MapLibre GL JS

Set mapRuntime: 'maplibre' on each layer

npm install maplibre-gl

This package

npm install mapbox-exif-layer

GeoTIFF sources optional

Only for .tif / .tiff URLs — JPEG-only setups do not need this.

npm install geotiff

Import

import { ParticleMotion, SmoothRaster } from 'mapbox-exif-layer';
